# Техническое задание: Ghost Layout Fixer

## 1. Общая информация

### 1.1 Название
Ghost Layout Fixer

### 1.2 Назначение
Разработать консольную утилиту для Windows 10/11, предназначенную для обнаружения и удаления фантомных раскладок клавиатуры, которые:
- отображаются в переключателе языка;
- отсутствуют в обычных настройках Windows;
- могут возвращаться после перезагрузки.

### 1.3 Основной сценарий
Первый поддерживаемый сценарий:
- в системе присутствует фантомная раскладка `English (United Kingdom)` / `en-GB`;
- пользователь хочет удалить её штатно и дочистить остатки в реестре.

---

## 2. Требования к продукту

### 2.1 Тип приложения
- CLI-утилита
- локальный запуск на Windows
- без установки службы

### 2.2 Целевая платформа
- Windows 10
- Windows 11

### 2.3 Язык реализации
- C++

### 2.4 Предполагаемые средства
Допустимо использовать:
- WinAPI
- PowerShell через запуск внешних команд
- стандартную библиотеку C++
- при необходимости небольшую библиотеку для аргументов командной строки

---

## 3. Функциональные требования

## 3.1 Команда `scan`
Утилита должна:
1. проверить наличие целевой раскладки в системных настройках и/или связанных данных;
2. просканировать известные ветки реестра;
3. вывести список найденных совпадений;
4. показать итоговый статус:
   - следы не найдены;
   - найдены следы;
   - обнаружена возможная фантомная раскладка.

### 3.1.1 Формат вывода
Вывод должен содержать:
- имя команды;
- целевую раскладку;
- список проверенных мест;
- найденные значения;
- итог.

---

## 3.2 Команда `fix --layout <code>`
Для MVP обязательно поддержать:
- `en-GB`

Команда должна:
1. проверить права запуска;
2. создать backup изменяемых веток реестра;
3. при необходимости выполнить штатное добавление раскладки;
4. выполнить штатное удаление раскладки;
5. удалить остаточные записи в известных ветках реестра;
6. вывести итоговый отчёт;
7. сообщить, что рекомендуется перезагрузка или выход из системы.

### 3.2.1 Поведение при ошибке
Если один из этапов завершается ошибкой:
- утилита должна зафиксировать ошибку в отчёте;
- не скрывать частично выполненные действия;
- завершиться с ненулевым кодом возврата, если фиксация не завершена успешно.

---

## 3.3 Команда `fix --layout <code> --dry-run`
Утилита должна:
- не изменять систему;
- не удалять записи;
- не вызывать запись в реестр;
- показывать планируемые действия.

---

## 3.4 Команда `backup`
Утилита должна:
- экспортировать затрагиваемые ветки реестра в `.reg` файл;
- сообщать путь к созданному backup.

---

## 3.5 Команда `restore --file <path>`
Утилита должна:
- проверить существование файла;
- выполнить импорт `.reg`;
- вывести результат операции.

---

## 4. Проверяемые ветки реестра

Для MVP обязательно реализовать работу со следующими ветками:

- `HKEY_CURRENT_USER\Keyboard Layout\Preload`
- `HKEY_CURRENT_USER\Keyboard Layout\Substitutes`
- `HKEY_USERS\.DEFAULT\Keyboard Layout\Preload`

Дополнительно допускается расширение списка, но только через белый список в коде.

---

## 5. Нефункциональные требования

### 5.1 Безопасность
- перед любыми изменениями должен создаваться backup;
- пользователь должен явно подтвердить выполнение изменений;
- должен быть режим `--dry-run`;
- запрещено удалять записи, не относящиеся к выбранной раскладке.

### 5.2 Логирование
Утилита должна логировать:
- запуск команды;
- проверенные ветки;
- найденные значения;
- выполненные изменения;
- ошибки;
- путь к backup.

Лог допускается в консоль.
Файловый лог можно отложить на post-MVP.

### 5.3 Надёжность
- утилита не должна аварийно завершаться на отсутствии ветки/ключа;
- отсутствие целевого ключа должно считаться штатной ситуацией;
- все ошибки внешних команд должны быть отражены в отчёте.

### 5.4 Поддерживаемость
Код должен быть разделён по ответственностям:
- парсинг аргументов;
- работа с реестром;
- запуск системных команд;
- бизнес-логика фикса;
- отчётность.

---

## 6. Предлагаемая архитектура

## 6.1 Модули
### `CliParser`
Разбор аргументов командной строки.

### `RegistryService`
- чтение веток;
- поиск совпадений;
- удаление значений;
- экспорт backup.

### `SystemCommandRunner`
- запуск `reg.exe`;
- запуск PowerShell-команд;
- получение exit code и stdout/stderr.

### `LayoutFixService`
Основная логика сценария:
- scan;
- dry-run;
- fix;
- backup;
- restore.

### `ReportPrinter`
Форматированный вывод результатов в консоль.

---

## 7. Сценарий работы команды fix

### 7.1 Последовательность
1. старт команды;
2. валидация аргументов;
3. проверка прав администратора;
4. scan текущего состояния;
5. создание backup;
6. добавление раскладки штатным способом;
7. удаление раскладки штатным способом;
8. очистка остатков в реестре;
9. повторный scan;
10. печать отчёта;
11. завершение с корректным кодом возврата.

### 7.2 Коды возврата
Рекомендуется:
- `0` — успех;
- `1` — общая ошибка выполнения;
- `2` — недостаточно прав;
- `3` — ошибка backup;
- `4` — ошибка применения fix;
- `5` — ошибка restore.

---

## 8. Требования к CLI

## 8.1 Поддерживаемые команды
```bash
ghost-layout-fixer scan
ghost-layout-fixer fix --layout en-GB
ghost-layout-fixer fix --layout en-GB --dry-run
ghost-layout-fixer backup
ghost-layout-fixer restore --file backup.reg
```

## 8.2 Требования к сообщениям
Сообщения должны быть:
- короткими;
- понятными;
- без лишней воды;
- пригодными для копирования в issue или README.

---

## 9. Ограничения первой версии
Первая версия не обязана:
- исправлять все виды языковых проблем Windows;
- поддерживать GUI;
- работать как фоновый агент;
- автоматически определять все экзотические раскладки;
- исправлять повреждённые языковые профили за пределами белого списка.

---

## 10. Тестирование

## 10.1 Минимальные тесты
Нужно проверить:
1. запуск `scan` без ошибок;
2. корректную работу `--dry-run`;
3. создание backup;
4. корректную обработку отсутствующих ключей;
5. защиту от запуска без прав;
6. корректную обработку неподдерживаемой раскладки.

## 10.2 Ручной сценарий приёмки
1. на тестовой системе воспроизвести кейс с `en-GB`;
2. выполнить `scan`;
3. выполнить `fix --layout en-GB --dry-run`;
4. выполнить `fix --layout en-GB`;
5. перезагрузить систему;
6. убедиться, что фантомная раскладка не вернулась;
7. при необходимости проверить `restore`.

---

## 11. Критерии приёмки
Работа считается принятой, если:
- утилита собирается без критических проблем;
- команды работают по ТЗ;
- backup создаётся до изменений;
- dry-run ничего не меняет;
- целевой кейс с фантомным `en-GB` исправляется на тестовой машине;
- итоговый отчёт понятен пользователю.

---
